# v1.1 — Sparkle Auto-Updates + Per-App Icon

**Goal**: When a new version of an `.app` is tagged and pushed, end users get an in-app update prompt and can install the new version with one click. The update is signed and verified end-to-end via Sparkle. Each `.app` also gets a custom icon (not the system default).

**Stack decision**: [Sparkle 2.x](https://sparkle-project.org/) (Swift Package Manager). It's the industry-standard macOS auto-update framework, supports EdDSA signatures, appcast.xml, and install-on-quit.

**Threat model**: a man-in-the-middle could replace our appcast.xml with a malicious one. The EdDSA signature on the `.dmg` URL prevents that — even if the appcast is tampered with, the signature won't match the public key baked into the `.app`, and Sparkle will reject the update.

## Phase 1: EdDSA Keypair (offline key gen)

**Task 1.1**: Generate a long-lived Ed25519 keypair for signing updates.

- **Tools**: `tools/generate_keys.py` (calls `cryptography` Python lib) or `openssl` + Sparkle's `generate_keys` CLI.
- **Output**:
  - `keys/odysseus_update_private.pem` — **NEVER committed**, lives in GitHub Actions secret `ODYSSEUS_UPDATE_PRIVATE_KEY_PEM`.
  - `keys/odysseus_update_public.txt` — base64-encoded public key (Sparkle's format), committed to the repo.
- **Acceptance criteria**:
  - [ ] `tools/generate_keys.py` is runnable and reproducible
  - [ ] Private key is added to `.gitignore` AND to GitHub repo secrets (or local keystore for testing)
  - [ ] Public key file is at `keys/odysseus_update_public.txt` and tracked in git
  - [ ] Re-running the script with the same seed produces the same keypair (deterministic for testing)
  - [ ] Public key is base64-encoded, 32 bytes, decodable

## Phase 5: Appcast generation + signing

**Task 5.1**: Create `appcasts/odysseus.xml.tmpl` (Sparkle appcast template).

- **Why**: Sparkle parses `appcast.xml` to know about new releases. Each `<item>` is one version.
- **Format** (Sparkle 2.x):
  ```xml
  <?xml version="1.0" encoding="utf-8"?>
  <rss version="2.0" xmlns:sparkle="http://www.andymatuschak.org/xml-namespaces/sparkle">
    <channel>
      <title>Odysseus</title>
      <item>
        <title>Odysseus 0.3.0</title>
        <pubDate>Mon, 02 Jun 2026 12:00:00 +0000</pubDate>
        <sparkle:version>0.3.0</sparkle:version>
        <sparkle:shortVersionString>0.3.0</sparkle:shortVersionString>
        <sparkle:minimumSystemVersion>11.0</sparkle:minimumSystemVersion>
        <enclosure url="https://github.com/.../Odysseus-0.3.0.dmg"
                   sparkle:edSignature="<base64>"
                   length="284567890"
                   type="application/x-apple-diskimage" />
      </item>
    </channel>
  </rss>
  ```
- **Acceptance criteria**:
  - [ ] `appcasts/odysseus.xml.tmpl` exists and renders valid XML
  - [ ] It contains the schema with `<title>`, `<item>`, and `<enclosure>` elements
  - [ ] It has placeholders for `${VERSION}`, `${PUB_DATE}`, `${DMG_URL}`, `${ED_SIGNATURE}`, `${DMG_LENGTH}`

**Task 5.2**: Create `tools/sign_update.py` — signs a `.dmg` with the private key.

- **Inputs**:
  - `--dmg <path>`: path to the .dmg
  - `--private-key <path>`: PEM-encoded Ed25519 private key
- **Outputs**:
  - Prints the base64-encoded Ed25519 signature of the .dmg file contents
- **Acceptance criteria**:
  - [ ] Reads the .dmg as bytes, signs with Ed25519, outputs base64 of the 64-byte signature
  - [ ] Verifiable: a test script that signs a known file and verifies with the public key returns True
  - [ ] Handles large files (200MB+) without OOM — reads in chunks or in one go (the .dmg is small enough to fit in memory, but stream-friendly is good)

**Task 5.3**: Update `ci/publish-release.sh` to produce the appcast.

- **Inputs**:
  - `<app-name>` (e.g. `odysseus`)
  - `<version>` (e.g. `0.3.0`)
- **Flow**:
  1. Locate the `.dmg` in `dist/`
  2. Sign the `.dmg` with the private key (from `keys/<app>_update_private.pem` or env var)
  3. Compute the `.dmg` length
  4. Render `appcasts/<app>.xml` from the template, inserting the new item
  5. Write `appcasts/<app>.xml` to disk
  6. Output the path to the rendered appcast + signed dmg
- **Acceptance criteria**:
  - [ ] Running `ci/publish-release.sh odysseus 0.3.0` produces `appcasts/odysseus.xml` with the new release
  - [ ] The new release's `sparkle:edSignature` is the base64 signature of the `.dmg`
  - [ ] The XML is valid (xmllint --noout)
  - [ ] Existing releases in the appcast are preserved (script reads existing appcast, appends new item)

## Phase 6: Per-app icon

**Task 6.1**: Design an Odysseus logo.

- **Concept**: Stylized boat/odysseus motif on a dark blue background (matching the app's dark theme). A simple but distinctive silhouette: triangular sail (cyan), curved hull (white/light gray), waves (subtle dotted pattern in the background). Square format, 1024x1024 source.
- **Files**:
  - `apps/odysseus/icon-1024.png` — source PNG (1024x1024)
  - `apps/odysseus/icon.icns` — generated by `iconutil` from a multi-size `.iconset/` directory
- **Acceptance criteria**:
  - [ ] The icon is visually distinct from the system default and from common app icons
  - [ ] It renders well at all macOS icon sizes (16, 32, 64, 128, 256, 512, 1024)
  - [ ] It's square, with a subject that's centered and doesn't bleed into corners (macOS rounds icons)

**Task 6.2**: Generate `icon.icns` from a `.iconset/`.

- **Why**: macOS uses multi-size `.icns` files that contain every required size.
- **Tool**: `iconutil` (built into macOS).
- **Flow**:
  1. Resize `icon-1024.png` to {16, 32, 64, 128, 256, 512, 1024}x{...}@1x and @2x sizes
  2. Put them in `apps/odysseus/icon.iconset/`
  3. Run `iconutil -c icns apps/odysseus/icon.iconset -o apps/odysseus/icon.icns`
- **Acceptance criteria**:
  - [ ] `apps/odysseus/icon.icns` exists and is a valid macOS icns file
  - [ ] `file apps/odysseus/icon.icns` reports "Mac OS X icon image"
  - [ ] `sips -g all apps/odysseus/icon.icns` shows multiple sizes

**Task 6.3**: Wire the icon into the build pipeline.

- **Why**: The build must copy `apps/<name>/icon.icns` into `<App>.app/Contents/Resources/icon.icns` (matching `CFBundleIconFile`).
- **Files**: `ci/build-app.sh`
- **Acceptance criteria**:
  - [ ] If `apps/<name>/icon.icns` exists, it's copied to `<App>.app/Contents/Resources/`
  - [ ] If it doesn't exist, the build still succeeds (falls back to system default; logs a warning)
  - [ ] `Info.plist.tmpl`'s `CFBundleIconFile` is already `icon` (no change needed)
  - [ ] The icon shows up in the Dock when the .app is launched

## Phase 8: End-to-end testing

**Task 8.1**: Build a v0.2.1 (clean) and a v0.3.0 (with feature) and verify the update path.

- **Steps**:
  1. Build and install v0.2.1 from `apps/odysseus/` (with version 0.2.1 in `webappify.yaml`)
  2. Verify the .app shows "Odysseus 0.2.1" in the About menu
  3. Build a v0.3.0 (with some new feature, e.g. new menu item, updated icon, etc.)
  4. Run `ci/publish-release.sh odysseus 0.3.0` to sign + create the appcast
  5. Serve the appcast + .dmg via a local HTTP server on port 9999
  6. Update the installed v0.2.1's `update_feed` to `http://127.0.0.1:9999/appcast.xml` — wait, this requires a rebuild. So instead, point `update_feed` at the local server from the start.
  7. Launch the installed v0.2.1, click Help → "Check for Updates…"
  8. Verify: Sparkle finds the v0.3.0 update, prompts user, downloads, installs on quit
  9. Relaunch, verify the .app is now v0.3.0

- **Acceptance criteria**:
  - [ ] v0.2.1 builds and installs cleanly
  - [ ] About menu shows "Odysseus 0.2.1"
  - [ ] Custom Odysseus icon shows in the Dock
  - [ ] `ci/publish-release.sh odysseus 0.3.0` produces a valid `appcasts/odysseus.xml`
  - [ ] Local HTTP server serves the appcast and .dmg
  - [ ] v0.2.1's "Check for Updates…" finds v0.3.0
  - [ ] Sparkle shows the update UI ("Odysseus 0.3.0 is available…")
  - [ ] User clicks Install, Sparkle downloads the .dmg
  - [ ] Signature verifies (Sparkle would log a verification error if not)
  - [ ] User quits the app, Sparkle installs the update
  - [ ] Relaunching shows v0.3.0
  - [ ] All other tests (Cardinal Rule, lift-and-shift, dark page renders) still pass

**Task 8.2**: Add an automated E2E test script `ci/test-update.sh`.

- **Why**: A non-interactive test that proves the update path works headless.
- **Approach**:
  - Build v0.2.1 with `update_feed` pointing to a local server
  - Serve appcast + .dmg from a local server
  - Run the v0.2.1 .app headless (server only, no GUI)
  - Trigger Sparkle's `checkForUpdates` programmatically (via a test mode flag or by running the binary with `--check-for-updates`)
  - Assert that the update is detected (via a log line in `wrapper.log` or a custom signal)
  - Don't actually install in the automated test (would require user interaction); just verify discovery
- **Acceptance criteria**:
  - [ ] `ci/test-update.sh` exits 0 when the update is discovered
  - [ ] The script logs the discovered version (e.g. "found update v0.3.0")
  - [ ] Running the script with a non-matching appcast (e.g. empty or with the same version) exits 0 but logs "no update found"

## Risks and how I'm handling them

| Risk | Mitigation |
|---|---|
| Sparkle SwiftPM version mismatch | Pin to a known-good version (e.g. 2.7.0+), test `swift package show-dependencies` in CI |
| macOS 14 runner might not have latest Sparkle | Use `macos-14` runner, which we already use; verify Sparkle builds on it before relying on it |
| EdDSA key handling in CI | Store the private key in a GitHub Actions secret, read with `secrets.ODYSSEUS_UPDATE_PRIVATE_KEY_PEM`, write to a temp file just for the signing step, never echo or log |
| Sparkle prompt blocking during first-run wizard | Defer `Updater.start()` until after first-run setup is complete |
| Sparkle 2.x needs `LSUIElement=false` in Info.plist (it isn't a background app) | Already false by default for our wrapper |
| Local HTTP server for testing isn't a real HTTPS endpoint | Test the discovery flow; document that production should use HTTPS. Sparkle will warn about non-HTTPS feeds but still works in dev. |
| `appcast.xml` schema drift | Pin to Sparkle 2.x schema; use the official `<rss version="2.0" xmlns:sparkle="...">` namespace |
